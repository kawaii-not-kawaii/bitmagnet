# Decision Log

## 2026-06-15

### D001: Separate LLM classifier worker (not inline)
- **Context:** Should the LLM step be inline in the default workflow (blocking classifier pipeline) or a separate background worker?
- **Decision:** Separate queue handler `llm_classify` running as a background worker alongside the normal `classify` handler. The normal classier marks unknowns as unmatched; the LLM worker picks them up asynchronously.
- **Rationale:** LLM classification is orders of magnitude slower than YAML+CEL (100ms vs 100ms+). Blocking the fast pipeline on LLM calls would create a bottleneck. A dedicated worker lets the normal pipeline flow unimpeded while the LLM chews through unknowns at its own pace.

### D002: TMDB enrichment after LLM classification
- **Context:** After LLM classifies content, should we also try TMDB?
- **Decision:** Yes. The LLM resolves title + year, then `attach_tmdb_content_by_search` fires with those values. If TMDB fails, the LLM's classification still stands.
- **Rationale:** TMDB enrichment is the difference between a classified torrent and a fully searchable one with poster, overview, etc.

### D003: Gemma 4 26B QAT as primary model
- **Context:** Which LLM should power classification?
- **Decision:** Gemma 4 26B QAT on `miniscoffee:8082` (100.125.213.44). Falls back to LFM 8B on `miniscoffee:8081` if primary unreachable.
- **Rationale:** 26B MoE (4B active) provides strong accuracy for classification tasks while running at ~100 tok/s on the available hardware. LFM 8B as backup.

### D004: Background worker model (not cache)
- **Context:** How to process unknowns without redundant work?
- **Decision:** The LLM runs as a persistent background queue worker. When the normal classifier leaves a torrent unmatched, it's re-queued to the `llm_classify` handler. The job queue's existing fingerprint dedup prevents duplicate processing. No additional cache layer needed.
- **Rationale:** The normal processor and LLM processor run side by side — one handles fast-path classification, the other picks up strays. Queue dedup is free, persistent, and crash-safe.
### D004: Background worker model (not cache)
- **Context:** How to process unknowns without redundant work or per-torrent cache?
- **Decision:** The LLM runs as a persistent background queue worker consuming from a dedicated `llm_classify` handler. The job queue's existing fingerprint dedup prevents duplicate processing. No additional cache layer.
- **Rationale:** The normal processor and LLM processor run side by side — one handles fast-path YAML+CEL classification, the other picks up strays asynchronously. Dedup is free and crash-safe via the job queue.

### D005: Per-request context budget
- **Context:** Gemma 4 has 256K context. A batch of torrents with file lists could eat it up.
- **Decision:** Configurable `max_context_tokens` (default 16K) caps total prompt per request. File lists truncated to fit. `max_tokens` caps output (256). Batch polling on a configurable `interval` (default 5s).
- **Rationale:** Each request to llama.cpp is stateless — no history accumulates between requests. Context budget prevents exceeding slot capacity deterministically.

### D006: Concurrent slot partitioning
- **Context:** Single slot means serial LLM processing. With 4+ bitmagnet workers, we want true parallelism.
- **Decision:** llama.cpp serves with `--parallel N` (N=4, matching `worker_count`). Total model context (256K) is shared across N slots via llama.cpp's internal slot scheduler. Each slot effectively gets ~64K. Bitmagnet workers each consume from the queue independently.
- **Rationale:** More slots = more concurrency without additional VRAM cost (same total KV cache). 64K per slot is still generous for batched torrent classification. Continuous batching (`--cont-batching`) handles dynamic load across slots.
